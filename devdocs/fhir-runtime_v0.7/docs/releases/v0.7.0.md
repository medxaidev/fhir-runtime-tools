# fhir-runtime v0.7.0

Release date: 2026-03-10

- `parseSearchParameter()` / `parseSearchParametersFromBundle()` — typed SearchParameter resource parsing
- `extractSearchValues()` / `extractAllSearchValues()` — FHIRPath-based search index value extraction
- `extractReferences()` / `extractReferencesFromBundle()` — Reference tree walking with classification
- `validateReferenceTargets()` — Reference target type validation against profile constraints
- `buildCapabilityFragment()` — CapabilityStatement REST fragment generation from profiles and search parameters

- 110 new tests — 3,376 total across 88 test files

## Added

### SearchParameter Parser

- `parseSearchParameter(json)` — parse a single SearchParameter JSON into typed `SearchParameter` object
- `parseSearchParametersFromBundle(bundle)` — batch parse SearchParameters from a FHIR Bundle
- Validates all required fields: `url`, `name`, `status`, `code`, `base`, `type`
- Supports all 9 search parameter types: number, date, string, token, reference, composite, quantity, uri, special
- Parses optional fields: `expression`, `description`, `target`, `modifier`, `chain`, `multipleOr`, `multipleAnd`, `version`

### Search Value Extractor

- `extractSearchValues(resource, searchParam)` — extract search index values for a single SearchParameter
- `extractAllSearchValues(resource, searchParams)` — extract values for all applicable SearchParameters
- Uses FHIRPath engine (`evalFhirPath`) for expression evaluation
- Type-specific extraction:
  - `string` — HumanName (family + given), Address (line + city + state + postalCode), plain string
  - `token` — Coding (system|code), CodeableConcept (first coding), Identifier (system|value), boolean
  - `reference` — literal/absolute reference parsing with resourceType and id extraction
  - `date` — date, dateTime, Period.start
  - `number` — numeric values
  - `quantity` — Quantity (value, unit, system, code)
  - `uri` — URI strings

### Reference Extractor

- `extractReferences(resource)` — walk resource tree and extract all Reference elements
- `extractReferencesFromBundle(bundle)` — extract References from all Bundle entries
- `validateReferenceTargets(resource, profile)` — validate reference target types against profile constraints
- Reference classification:
  - `literal` — relative references (e.g., `Patient/123`)
  - `absolute` — full URL or URN references (e.g., `https://example.org/fhir/Patient/456`)
  - `contained` — fragment references (e.g., `#contained-1`)
  - `logical` — identifier-based references (system|value)
- Extracts `targetType`, `targetId`, `display` where inferrable

### CapabilityStatement Builder

- `buildCapabilityFragment(profiles, searchParams?, mode?)` — generate CapabilityStatement REST fragment
- Groups profiles by resource type with `profile` (specialization) and `supportedProfile` (constraint) separation
- Attaches applicable search parameters to each resource entry (including `Resource`-level params)
- Deduplicates search parameters by code
- Filters retired search parameters
- Sorts resources alphabetically, search parameters alphabetically

### Resource Type Registry

- `ResourceTypeRegistry` class — registry of known FHIR resource types
  - `register()` / `get()` / `isKnown()` / `remove()` / `clear()`
  - `getAll()` / `getConcreteTypes()` (filters out abstract types)
  - `size` property
  - `static fromContext(context)` — build from FhirContext
  - `static fromList(types)` — build from array
- `FHIR_R4_RESOURCE_TYPES` constant — 148 FHIR R4 resource types

### Type Definitions

- 12 new type exports: `SearchParamType`, `SearchParameter`, `SearchIndexValue`, `SearchIndexEntry`, `ReferenceType`, `ReferenceInfo`, `CapabilitySearchParam`, `CapabilityRestResource`, `CapabilityStatementRest`, `ResourceTypeInfo`

### Public API Surface

- 12 type exports + 11 value exports added to `src/index.ts` from integration module
- Total public exports: ~280+ symbols across 11 modules

## Notes

- `extractSearchValues()` uses the existing FHIRPath engine for expression evaluation — no new parser needed
- No SQL generation or persistence logic — fhir-runtime handles parsing and extraction only
- `ResourceTypeRegistry.fromContext()` builds registry from FhirContext with loaded StructureDefinitions
- Version bumped to v0.7.0 (not v1.0-rc) — comprehensive evaluation and testing required before API freeze
- fhir-persistence can now begin formal development using the integration module's contracts
- This release remains fully backward compatible with v0.6.0
