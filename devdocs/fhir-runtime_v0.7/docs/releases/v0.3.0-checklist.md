- [x] `package.json` version is `0.3.0`
- [x] `CHANGELOG.md` contains `0.3.0` release notes
- [x] `README.md` reflects `v0.3.0`
- [x] `docs/api/fhir-runtime-api-v0.3.md` exists
- [x] `docs/specs/engine-capability-contract-v0.3.md` exists
- [x] `docs/releases/v0.3.0.md` exists

## 2. Documentation Consistency

- [x] Release-facing docs use `Provider Abstraction Layer (STAGE-1)` terminology
- [x] Release-facing docs use `OperationOutcomeBuilder` terminology
- [x] Release-facing docs describe `v0.3.0` as backward compatible with `v0.2.0`
- [x] Release-facing docs describe `STAGE-2 (v0.4.0)` as the next step for actual terminology validation
- [x] README links point to the `v0.3` API and capability documents

## 3. Test & Build Gates

- [ ] Run `npm test`
- [ ] Confirm provider-focused tests remain green
- [ ] Run `npm run build`
- [ ] Confirm `dist/` is generated successfully
- [ ] Confirm generated type declarations and module entrypoints match `package.json`

## 4. Package Publish Artifact Verification

Expected published package contents based on `package.json`:

- `dist/`
- `README.md`
- `LICENSE`
- `CHANGELOG.md`
- `package.json` (included by npm and explicitly exported)

Verification points:

- [x] `files` field includes only publishable runtime artifacts and root metadata
- [x] npm package is not expected to include `docs/`, `devdocs/`, `src/`, or `spec/`
- [x] `main` points to `./dist/cjs/index.cjs`
- [x] `module` points to `./dist/esm/index.mjs`
- [x] `types` points to `./dist/index.d.ts`
- [x] `exports["."]` defines both `import` and `require` entrypoints
- [x] `exports["./package.json"]` is present
- [x] `prepublishOnly` runs `npm run test && npm run build`

## 5. Pre-Publish Sanity Checks

- [ ] Run `npm pack --dry-run`
- [ ] Verify tarball contents include expected `dist` outputs and root docs only
- [ ] Verify no unexpected large source/test/doc folders are included in the tarball
- [ ] Verify generated package size is reasonable

## 6. Git Release Steps

- [ ] Review final git diff
- [ ] Commit release prep changes
- [ ] Create git tag `v0.3.0`
- [ ] Push branch and tag
- [ ] Create GitHub Release using `docs/releases/v0.3.0.md`

## 7. npm Publish Steps

- [ ] Ensure npm authentication is ready
- [ ] Publish package to npm
- [ ] Verify `fhir-runtime@0.3.0` appears on npm registry
- [ ] Verify README renders correctly on npm package page

## 8. Post-Release Verification

- [ ] Install published package in a clean sample project
- [ ] Verify ESM import works
- [ ] Verify CJS require works
- [ ] Verify type declarations resolve correctly
- [ ] Verify provider exports are available from top-level package

## 9. Known Non-Blocking Notes

- Repository documentation under `docs/` is intentionally richer than the npm publish artifact set
- `docs/api/`, `docs/specs/`, and `docs/releases/` are repository documentation and are not expected to ship in the npm package
- Actual terminology validation remains out of scope for `v0.3.0` and is planned for `STAGE-2 (v0.4.0)`
