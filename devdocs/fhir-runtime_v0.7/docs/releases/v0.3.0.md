# fhir-runtime v0.3.0

Release date: 2026-03-07

## Highlights

- Provider Abstraction Layer (`STAGE-1`) released
- New public contracts: `TerminologyProvider` and `ReferenceResolver`
- New default implementations: `NoOpTerminologyProvider` and `NoOpReferenceResolver`
- New `OperationOutcomeBuilder` helpers for validation, parse, and snapshot result conversion
- Backward compatible validator integration through optional provider hooks
- Ready to begin downstream `fhir-server` integration work before full terminology support lands in `v0.4.0`

## Added

### Provider Abstraction Layer (STAGE-1)

- New `src/provider/` module added to the public package surface
- Added `TerminologyProvider` interface with:
  - `validateCode()`
  - `expandValueSet()`
  - `lookupCode()`
- Added `ReferenceResolver` interface with:
  - `resolve()`
  - `exists()`
- Added `NoOpTerminologyProvider`
- Added `NoOpReferenceResolver`
- Added `buildOperationOutcome()` as part of the `OperationOutcomeBuilder` surface
- Added `buildOperationOutcomeFromParse()` as part of the `OperationOutcomeBuilder` surface
- Added `buildOperationOutcomeFromSnapshot()` as part of the `OperationOutcomeBuilder` surface
- Added `OperationOutcome`, `OperationOutcomeIssue`, and `OperationOutcomeIssueType` types

### Validator Integration

- Extended `ValidationOptions` with optional `terminologyProvider`
- Extended `ValidationOptions` with optional `referenceResolver`
- Preserved existing behavior when providers are not supplied

## Testing

- Added 97 new tests across 6 provider-focused test files
- Verified interface contracts for terminology and reference abstractions
- Verified NoOp default behavior for backward compatible integration
- Verified `OperationOutcomeBuilder` conversion from validation, parse, and snapshot flows
- Verified validator integration points with provider-aware options
- Confirmed existing v0.2.0 test baseline remains passing

## Compatibility

- No breaking API changes from `v0.2.0`
- Provider integration points are optional
- Existing validation workflows remain backward compatible without external terminology or reference services

## Upgrade Notes

No migration changes are required for existing `v0.2.0` consumers.

You can continue using the package as before:

```bash
npm install fhir-runtime
```

If you are integrating with a server or platform layer, you can now start implementing:

- `TerminologyProvider`
- `ReferenceResolver`

## Known Limits

- Real terminology binding validation is not included in this release
- Provider APIs are async by design, but full terminology-driven validation behavior is planned for `v0.4.0`
- IG package loading and advanced canonical resolution remain planned for later stages

## Next Step

Planned next release: `v0.4.0`

Focus areas:

- Binding strength validation
- In-memory terminology provider
- CodeSystem / ValueSet registries
- Inline ValueSet membership checks
- US Core terminology integration tests
