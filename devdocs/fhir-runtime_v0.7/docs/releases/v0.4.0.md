# fhir-runtime v0.4.0

Release date: 2026-03-08

## Highlights

- Validation Pipeline & DX Enhancement (`STAGE-4`) released
- New composable `ValidationPipeline` with pluggable step architecture
- Three built-in validation steps: `StructuralValidationStep`, `TerminologyValidationStep`, `InvariantValidationStep`
- Hook system for pipeline lifecycle events (`beforeValidation`, `afterValidation`, `beforeStep`, `afterStep`, `onIssue`, `onError`)
- Batch validation for multiple resources in a single pipeline run
- Enhanced error messages with fix suggestions and documentation links
- Structured validation reports with issue grouping by severity, path, and step
- Backward compatible — existing `StructureValidator` usage remains valid

## Added

### Validation Pipeline (STAGE-4)

- New `src/pipeline/` module added to the public package surface
- Added `ValidationPipeline` class with:
  - `addStep()` / `removeStep()` — composable step management
  - `validate()` — single resource validation
  - `validateBatch()` — batch validation of multiple resources
  - `on()` — hook registration for lifecycle events
  - `getSteps()` — sorted step inspection
- Added `ValidationStep` interface — pluggable validation step contract
- Added `PipelineContext` interface — shared state passed between steps
- Added `PipelineOptions` — pipeline configuration (`failFast`, `minSeverity`, `maxDepth`, providers)

### Built-in Validation Steps

- Added `StructuralValidationStep` — wraps existing `StructureValidator` (priority: 10)
- Added `TerminologyValidationStep` — validates coded elements via `TerminologyProvider` (priority: 20)
- Added `InvariantValidationStep` — evaluates FHIRPath constraints (priority: 30)

### Hook System

- Added `HookManager` class for registering and emitting async lifecycle events
- Added `PipelineEvent` type: `beforeValidation`, `afterValidation`, `beforeStep`, `afterStep`, `onIssue`, `onError`
- Added `PipelineEventHandler` and `PipelineEventData` types

### Batch Validation

- Added `BatchEntry` / `BatchResult` / `BatchEntryResult` types
- Sequential batch execution with per-entry result collection
- Labels for identifying entries in batch results

### Enhanced Error Messages

- Added `EnhancedValidationIssue` interface extending `ValidationIssue` with:
  - `suggestion` — actionable fix recommendation
  - `documentationUrl` — link to relevant FHIR spec section
  - `expected` / `actual` — structured comparison values
- Added `enhanceIssue()` and `enhanceIssues()` functions
- Enhancement rules for all 16+ `ValidationIssueCode` types

### Validation Reports

- Added `ValidationReport` interface with `ReportSummary`
- Added `generateReport()` function producing structured reports
- Issue grouping by severity, path, and step name

### New Exports

New top-level type exports in `v0.4.0`:

- `ValidationStep`, `PipelineContext`, `PipelineOptions`, `PipelineResult`, `StepResult`
- `BatchEntry`, `BatchResult`, `BatchEntryResult`
- `PipelineEvent`, `PipelineEventHandler`, `PipelineEventData`
- `EnhancedValidationIssue`, `ValidationReport`, `ReportSummary`

New top-level value exports in `v0.4.0`:

- `ValidationPipeline`
- `StructuralValidationStep`, `TerminologyValidationStep`, `InvariantValidationStep`
- `generateReport`
- `enhanceIssue`, `enhanceIssues`
- `HookManager`

## Upgrade Notes

No migration changes are required for existing `v0.3.0` consumers.

To adopt the new pipeline:

```typescript
import {
  ValidationPipeline,
  StructuralValidationStep,
  TerminologyValidationStep,
  InvariantValidationStep,
} from "fhir-runtime";

const pipeline = new ValidationPipeline({ failFast: true });
pipeline.addStep(new StructuralValidationStep());
pipeline.addStep(new TerminologyValidationStep());
pipeline.addStep(new InvariantValidationStep());

const result = await pipeline.validate(resource, profile);
```

## Known Limits

- Batch validation is sequential (no concurrency control)
- TerminologyValidationStep requires an external `TerminologyProvider` to perform real terminology checks
- Enhanced messages cover all standard issue codes; custom codes need manual enhancement
- Report generator produces in-memory reports (no file output)
