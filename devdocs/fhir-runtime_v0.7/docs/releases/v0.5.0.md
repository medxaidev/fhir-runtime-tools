# fhir-runtime v0.5.0

Release date: 2026-03-09

## Highlights

- Terminology Binding Validation (`STAGE-3`) released
- New `InMemoryTerminologyProvider` — fully functional local terminology validation
- Binding strength validation: `required` / `extensible` / `preferred` / `example`
- `CodeSystemRegistry` and `ValueSetRegistry` for in-memory terminology storage
- ValueSet membership evaluation: expansion, compose (include/exclude), hierarchical filters
- Full integration with existing `TerminologyValidationStep` in the pipeline
- Bundle loading support for CodeSystems and ValueSets
- Backward compatible — all 2,995 v0.4.0 tests still pass

## Added

### Terminology Module (STAGE-3)

- New `src/terminology/` module added to the public package surface
- Added `InMemoryTerminologyProvider` class implementing `TerminologyProvider`:
  - `registerCodeSystem()` / `registerValueSet()` — manual registration
  - `loadFromBundle()` — batch load from FHIR Bundle
  - `validateCode()` — code validation against CodeSystem or ValueSet
  - `expandValueSet()` — ValueSet expansion with filter and pagination
  - `lookupCode()` — code lookup with display and definition
- Added `validateBinding()` — core binding validation function
- Added `extractCodedValues()` — extract codes from FHIR coded elements

### Binding Strength

- Added `severityForBindingStrength()` — map strength to issue severity
- Added `severityWhenNoProvider()` — severity when no provider available
- Added `requiresValidation()` — whether binding needs validation
- Added `bindingStrengthDescription()` — human-readable strength description

### Registries

- Added `CodeSystemRegistry` — in-memory CodeSystem storage with:
  - Hierarchical concept lookup
  - `isDescendantOf()` for is-a relationship checks
  - `allCodes()` for flat code enumeration
- Added `ValueSetRegistry` — in-memory ValueSet storage

### ValueSet Membership

- Added `isCodeInValueSet()` — membership evaluation supporting:
  - Pre-expanded ValueSets (expansion.contains)
  - Compose-based ValueSets (include/exclude rules)
  - Enumerated concept lists
  - Filter operations: `is-a`, `is-not-a`, `in`, `not-in`, `=`, `regex`, `exists`
  - CodeSystem hierarchy traversal for `is-a` filters

### Type Definitions

- Added `CodeSystemDefinition`, `CodeSystemConcept`
- Added `ValueSetDefinition`, `ValueSetCompose`, `ValueSetComposeInclude`, `ValueSetComposeConcept`, `ValueSetComposeFilter`
- Added `ValueSetExpansionDef`, `ValueSetExpansionContainsDef`
- Added `BindingValidationResult`, `BindingConstraintInput`

## Compatibility

- All v0.4.0 public API remains unchanged
- `TerminologyValidationStep` (STAGE-2) now fully functional with `InMemoryTerminologyProvider`
- Zero runtime dependencies (unchanged)
- Node.js >=18.0.0 (unchanged)
- ESM primary + CJS compatibility (unchanged)

## Upgrade Notes

1. Install as usual — no breaking changes
2. Optionally create `InMemoryTerminologyProvider` and inject into `ValidationPipeline` or `ValidationOptions`
3. `TerminologyValidationStep` in the pipeline automatically uses the injected provider

## Known Limits

- `InMemoryTerminologyProvider` is not suitable for large CodeSystems (e.g., full SNOMED CT)
- ValueSet compose filter support is partial: `is-a` requires CodeSystem hierarchy in registry
- No remote terminology server support (by design — belongs to fhir-server layer)
